# 第1章 モノリシックアーキテクチャ
モノリシックアーキテクチャでは、ソフトウェアのすべてのコンポーネントが緊密に結合し、相互に依存しながら、単一の統一されたユニットを形成しています。このアプローチは、特に小規模なプロジェクトでは、シンプルで開発のしやすさを提供します。しかし、プロジェクトの規模が大きくなると、コンポーネントが相互に絡み合っているため、保守、更新、拡張が困難になることがあります。

- [「中世の城」 王国の栄光と衰退](./1_monolithic)

# 第2章 多層アーキテクチャ
多層アーキテクチャは、ソフトウェアコンポーネントを明確なレイヤに編成し、それぞれが責任を持つようにしたものです。一般的には、プレゼンテーション層、ビジネスロジック層、データアクセス層が存在します。このアプローチは、懸念事項の分離とモジュール化を促進し、保守と理解を容易にするものです。しかし、注意深く管理しないと、依存関係やパフォーマンスのボトルネックが発生する可能性があります。

- [「階層型都市」調和とバランスへの挑戦](./2_layered_architecture)

# 第3章 パイプラインアーキテクチャ
パイプラインアーキテクチャでは、ソフトウェアは一連の処理ステージとして構成され、データはパイプラインのようにその間を流れます。各ステージは、次のステージにデータを渡す前に、データに対して特定の処理を行います。このアーキテクチャは、並列処理やワークフローの合理化に適していますが、エラー処理やステージ間の同期に問題がある場合があります。

- [「複雑な水道橋」知識の流れ ](./3_pipeline_architecture)

# 第4章 マイクロカーネルアーキテクチャ
マイクロカーネルアーキテクチャは、最小限のコアシステムと、追加機能を提供する拡張可能なプラグインモジュールで構成されています。モジュールを変更してもコアシステムには影響しないため、柔軟性、モジュール性、メンテナンスの容易性が得られます。しかし、モジュール間の通信が必要なため、複雑で性能のオーバーヘッドが発生する可能性があります。

- [「モジュール工場」 中央処理装置による効率的な管理](./4_microkernel)

# 第5章 サービス指向アーキテクチャー（SOA）
SOAとは、ソフトウェアを疎結合で再利用可能なサービスの集合体として構成するアーキテクチャスタイルです。これらのサービスは、標準的なプロトコルで相互に通信し、タスクを実行します。SOAはモジュール化、再利用性、相互運用性を促進しますが、サービスのオーケストレーション、管理、発見において複雑さをもたらす可能性があります。

- [「列車ネットワーク」サービス駅と列車の関係](./5_service_oriented)

# 第6章 イベント駆動型アーキテクチャ（EDA）
EDAでは、ソフトウェアコンポーネントは、状態の変化を表す個別のメッセージであるイベントを介して通信します。コンポーネントは、直接結合することなく、イベントを生成、消費、反応できます。EDAは非同期、並列処理をサポートし、スケーラビリティと疎結合を促進します。しかし、イベントフローの管理、一貫性の確保、デバッグに課題が生じる可能性があります。

- [「お祭り」 イベントの祭典](./6_event_driven)

# 第7章 スペースベースアーキテクチャ
スペースベースアーキテクチャは、ソフトウェアコンポーネントを複数のノードに分散し、パーティショニング、レプリケーション、ルーティングに重点を置いてスケーラビリティとアベイラビリティを処理します。トラフィックが多く、高性能なアプリケーションの処理に適しています。ただし、分散システムの管理、同期、一貫性などに複雑さが生じる可能性があります。

- [「宇宙ステーション」 巨大な相互グリッド](./7_space-based)

# 第8章 オーケストレーション主導のサービス指向アーキテクチャ
このアーキテクチャは、サービスの相互作用とワークフローを管理する集中型オーケストレータでSOAを拡張します。このアプローチは、サービスの調整とエラー処理を簡素化しますが、単一障害点を生み出し、オーケストレーターの実装と管理に複雑さをもたらす可能性があります。

- [「オーケストラ」 マエストロのマスターピース ](./8_orchestration_driven)

# 第9章 マイクロサービスアーキテクチャ
マイクロサービスアーキテクチャは、ソフトウェアを小さく、独立して展開可能なサービスに分割し、それぞれが特定の機能を担当するものです。このアプローチは、スケーラビリティ、柔軟性、フォールトトレランスを促進します。しかし、サービスの管理、調整、展開に複雑さが生じます。

- [「列島」 独立した島の交流](./9_microservices)

# 第10章 モジュール型モノリス
モジュラーモノリスは、単一の統一されたシステム内で、ソフトウェアを独立した、明確に定義されたモジュールに編成します。このアプローチは、モノリシックアーキテクチャのシンプルさとマイクロサービスのモジュール性を併せ持ち、保守性と懸念事項の分離を促進します。しかし、モジュール間の依存関係や通信を管理する上で課題に直面することがあります。

- [「橋でつながれた都市」 コミュニティの協調](./10_modular_monolith)

# 第11章 サーバーレスアーキテクチャ
サーバーレスアーキテクチャでは、アプリケーションは、イベントに応じてオンデマンドで実行される小さなステートレス機能で構成されています。このアプローチは、スケーラビリティ、コスト効率、運用オーバーヘッドの削減を実現します。しかし、コールドスタートの管理、監視、デバッグに課題が生じる可能性があります。

- [「魔法の森」魔法しかけのエフェメラル](./11_serverless)

# 第12章 クリーンなアーキテクチャ
クリーンアーキテクチャは、懸念事項の分離と依存関係の逆転に焦点を当て、コンポーネントを明確な境界を持つ明確なレイヤーに編成します。このアプローチは、保守性、テスト性、および適応性を促進します。しかし、開発者に複雑さと学習曲線をもたらす可能性があります。

- [「時計塔」クリーンな建築のシンボル](./12_clean_architecture)

# 第13章 🧅オニオンアーキテクチャ
オニオンアーキテクチャもまた、懸念の分離と依存関係の逆転を強調しますが、コンポーネントを同心円状の層に構成し、最も内側の層にはコア・ビジネス・ロジックが含まれます。このアプローチは柔軟性とモジュール性を提供しますが、依存関係や抽象化を慎重に管理する必要があるかもしれません。

- [悟りの層 「古代の寺院」](./13_onion_architecture)
